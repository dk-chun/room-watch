# 방 매물 추적 (직방)

신논현·강남·양재·양재시민의숲역 통근권의 전세/월세 매물을 주기적으로 대조해
**"지난번 대비 뭐가 바뀌었나"**만 보여주는 스킬. 사용자는 삼성전자 통근(회사=양재시민의숲,
셔틀=강남 서초사옥·양재역 2번출구), 신논현 이북은 제외.

## 수집은 자동임 — 재수집하지 말 것

**매일 07:00 KST에 GitHub Actions가 알아서** 수집·diff·실거래대조·대시보드 갱신을 함.
대시보드: https://dk-chun.github.io/room-watch/ · repo: `dk-chun/room-watch`

사용자가 "매물 확인해줘" 하면 **이 순서로만** 하면 됨 (수초):

```
git -C <이 스킬 경로> pull        # CI가 커밋한 최신 스냅샷 + report.json 받기
```
1. `report.json` → 신규·빠짐·가격변동 파악
2. 최신 `snapshots/*.json`의 각 매물 `rt`(실거래) → **저평가 + 통근 좋은 매물 선별**(아래 precision)

⚠️ **`python fetch_listings.py`를 돌리지 말 것.** 5분 낭비인 데다, 로컬 스냅샷이 CI 스냅샷과 갈라져서 다음 diff가 통째로 깨짐.

### 즉시 갱신이 필요할 때만 (사용자가 "지금 막 올라온 것까지" 요구)
```
gh workflow run update.yml -R dk-chun/room-watch   # 5~18분 소요, 이후 git pull
```
로컬 실행이 정말 필요하면 반드시 `git pull` 후에 `python3 fetch_listings.py` (바탕화면 HTML로 나감).

## 실거래 대조 (recall=코드 / precision=사람)
매물 유형별로 그 건물/동네 실거래를 붙임. **코드는 넉넉히 다 붙이고(recall), 판단 필터는 안 넣음.**
- 오피스텔: pnu 지번 → 국토부 Offi 매칭 (건물명+실거래)
- 원룸-빌라/연립: 직방 `bls/rts` · 국토부 RH 지번 (건물)
- 원룸-다가구: 국토부 SH 동+면적 폴백 (지번 비공개라 동네시세)
- tier: `building`(건물) > `area`(동±5) > `area-wide`(동±15) > `dong`(동전체). 신뢰도 순.
- 전세 diff = 정확(보증금 직접), **월세 diff = 참고(전세환산이라 보증금 구조차로 거침)**.
- **precision은 여기(사람)서**: 붙은 rt를 보고 "실거래 대비 저평가 + 통근 좋은" 매물을 직접 추려 요약. 특히 월세는 보증금 구조 감안해 "진짜 싼 것 vs 보증금 낮아 그런 것" 구분. 코드로 자동 컷하지 말 것.

## 출력 해석 → 사용자 리포트

채팅엔 **변동 요약만 짧게**, 실제 매물 훑기는 대시보드로 유도. JSON을 읽어서 음슴체로:

- 항상 마지막에 **"바탕화면 `방매물.html` 열어보셈"** 안내(사진·정렬·필터 거기서).
- `baseline: true` → 첫 실행. "기준점 잡았고 대시보드 만들었음" 안내.
- `new` → 🆕 **신규 매물** 건수 + 눈에 띄는 것 1~2개만. 나머지는 대시보드에 🆕 초록으로 뜬다고.
- `removed` → ❌ **빠진 매물**(계약/내림). 건수 + 관심매물이 빠졌으면 짚어줄 것.
- `price_changed` → 💰 **가격 변동**. `before`→`after`. 내렸으면 집주인 급한 시그널이라 강조.
- 셋 다 비면 "새로 올라오거나 바뀐 거 없음"으로 끝.
- `dashboard_error`가 있으면 (C드라이브 접근 실패 등) 사용자에게 알릴 것.

채팅에 매물 표를 길게 나열하지 말 것 — 그게 대시보드를 만든 이유임.

## 네이버 부동산 (로컬 온디맨드 전용)

직방과 별개로 **네이버 매물**도 볼 수 있음. 사용자가 "네이버도 봐줘" 하면:

```
node <이 스킬 경로>/naver_fetch.mjs      # 5~8분, 바탕화면 방매물_네이버.html 생성
```

⚠️ **CI 자동화 불가.** GitHub Actions(Azure IP)는 네이버가 `net::ERR_CONNECTION_RESET`으로
네트워크 레벨 차단함(2026-07-10 검증). **로컬(가정용 IP)에서만** 됨. 크론에 넣지 말 것.
재검증하려면 `gh workflow run naver_check.yml -R dk-chun/room-watch`(수동, 실패가 정상).

- 방식: 헤드리스 크롬(CDP)으로 SPA가 쏘는 `Authorization` Bearer 토큰을 가로채 in-page fetch.
  curl은 무조건 429(TLS 지문차단)라 못 씀. node 내장 WebSocket만 사용(설치 의존성 0).
- 필터·통근 로직은 직방과 동일(전용 9평+, 최근접역 1.2km, 신논현 이북 제외).
  **가격 컷 없음(recall)** → 1000건 넘게 나옴, 15억 전세도 섞임. precision은 사람이.
- 네이버가 직방보다 데이터 풍부: **실좌표·건물명·방개수(상세API)·동일주소 매물수** 노출
  (직방은 다 흐려서 pnu로 우회해야 했음). 전세 인벤토리가 직방(37건)보다 압도적(400+건).
- 실거래 대조는 아직 미구현(v1). 필요하면 건물명/좌표로 붙이기 직방보다 쉬움.

## 조건 변경

`fetch_listings.py` 상단 CONFIG 수정: 역 좌표, `MIN_M2`(면적 하한), `MAX_DIST`(역 거리컷),
`EXCLUDE_NORTH_LAT`(신논현 이북 제외 위도; None이면 미적용). 예산/전월세 필터는 수집 단계엔 없고
리포트할 때 대화로 거르면 됨(스냅샷엔 전세·월세 전부 저장됨).

## 데이터

`snapshots/*.json` — 실행마다 누적되는 비교 기준. **지우지 말 것.** 파일명이 곧 실행 시각.
직방 웹 상세 링크는 헤드리스로 403이지만 브라우저에선 정상.
